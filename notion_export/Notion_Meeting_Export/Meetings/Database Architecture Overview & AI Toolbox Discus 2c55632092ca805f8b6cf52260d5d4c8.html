Summary

Steve's AI Solution for Data Engineering

  • Steve is developing a custom AI solution called "Fabric Toolbox" that integrates with Microsoft technologies to assist with data engineering tasks. [1] [2]
  • The solution can connect to databases, identify relevant tables, create joins, set up data pipelines, and build semantic models using natural language commands. [3] [4] [5]
  • It combines work from multiple open source projects and is customized specifically for IP Corporation's industry context. [6] [7] [8]
  • The tool includes guardrails to prevent impacting production databases and provides logs of all queries for security purposes. [9]
  • It's designed to work entirely within Microsoft's ecosystem, keeping all data in-house. [10] [11]

Current Database Environment

  • The company primarily uses Microsoft SQL Server for their databases. [12]
  • The main production server (M3DB1) hosts the ERP database called M3, which uses cryptic table naming conventions (e.g., "Kusma" for customer table). [12] [13]
  • Supporting databases include:
    • MDP (Metadata Publisher) - contains definitions for M3 database columns [14]
    • Document management database (on M3DB2) [15] [16]
    • Optiva (Formula 1 application) - manages product formulas, ingredients, specifications [17] [18]
    • Scheduling database - integrates with M3 [19]
    • ETQ database [20]
    • CSAID (Service Desk interface) - stores service desk requests [20] [21]
    • X-Rite - application for colorant data management [22] [23]
  • Most of these databases and applications integrate with the main M3 ERP system. [24]

Database Administration Practices

  • The database administrator handles maintenance tasks like reindexing databases, though BI and reporting responsibilities have reduced this focus. [25] [26]
  • SQL Server Agent is used to schedule stored procedures and jobs. [27] [28]
  • They rarely use triggers and cursors, preferring stored procedures and views for reporting needs. [29] [30] [31]
  • The admin occasionally performs manual database maintenance despite having automated weekly maintenance jobs. [26]
  • There's interest in developing an in-house solution to monitor database health and maintenance needs. [32] [33] [34]

Challenges in the Current Architecture

  • Reports are currently built directly against transactional databases rather than a proper data warehouse. [35] [36]
  • The cryptic table naming in M3 makes it difficult for users to understand the data structure without extensive knowledge. [13] [37] [38]
  • No formal initiative has been established to create a proper reporting layer, with solutions developed on an as-needed basis. [39] [40]
  • When pulling data from the ERP, fields must be renamed to be user-friendly (e.g., "Kono" to "customer number"). [41] [42]
  • There's uncertainty about how to handle the transition to a more structured data approach with bronze/silver layers. [42] [43]
