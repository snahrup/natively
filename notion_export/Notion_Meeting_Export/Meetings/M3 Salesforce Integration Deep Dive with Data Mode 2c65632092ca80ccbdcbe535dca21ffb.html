Introduction and Roles

  • Matt Francis was introduced as a developer who has worked on data migration and mass updates, now more aligned with data engineering ‣ ‣
  • Matt is expected to be more involved in the Master Data Management (MDM) process in the future ‣
  • The company currently lacks a Salesforce administrator as the previous person left and they are looking for a replacement ‣ ‣

M3 System Overview

  • M3 is the company's ERP system with a normalized database structure ‣
  • The system organizes data by company, division, and warehouse:
    • Company 700 is the current company ‣
    • Division 300 is for molding products, Division 400 is North American products ‣
    • Interplastic is in Division 100 and 200 ‣ ‣
    • Corporate is Division 001 ‣

Metadata Publisher Tool

  • The team demonstrated the Metadata Publisher tool which helps understand M3's database structure ‣ ‣
  • The tool is available in both cloud and on-premises versions with slight differences ‣ ‣ ‣
  • Access to the tool is through Azure ‣
  • Steve needs to get access to these systems, which will be arranged through Lori ‣ ‣

Data Structure in M3

  • The system uses 6-character table names that are difficult to understand without tools like Metadata Publisher ‣
  • Data in M3 is typically joined using compound keys:
    • Supplier data uses company number (KONO) and supplier number (SUNO) ‣
    • Additional scoping is provided by division (DIVI) and warehouse location (WHLO) ‣
  • The team demonstrated how to look up customer data using tables like OCUSMA ‣ ‣
  • MVX is the primary data schema ‣

Salesforce Integration

  • There's synchronization between M3 and Salesforce, primarily for customer data ‣ ‣
  • The team uses dbAMP (referred to as DBAM) to connect to Salesforce via SQL ‣ ‣ ‣
  • dbAMP was described as a good solution that puts integration "on your turf rather than theirs" ‣ ‣

Price Support Process

  • The system handles price supports where the company may offer discounts to distributors ‣
  • Price support tracking is managed separately from M3 ‣
  • The process involves:
    • Distributors submit price reports ‣
    • VSTOs (Visual Studio Tools for Office) were built to handle distributors' price reports ‣
    • At month end, distributors send items they were promised price supports on ‣
    • After credit approval, an email is sent with the information ‣ ‣
  • Sarah Wahlberg (business analyst and former customer service rep) might provide more information about this process ‣

Other Systems Mentioned

  • ETQ system was chosen over a Salesforce-based alternative, which IT may regret ‣ ‣
  • Matt handles synchronization of data from ERP to ETQ ‣

M3 Data Model & Normalization

  • Primary schema MVX; archive schema MVXARCH
  • Core masters: OCUSMA (customer), MITMAS (item) – split into related tables
  • Supplier tables linked by CONO + SUNO (company + supplier ID)
  • Warehouse/Division scope via WHLO and DIVI
  • Natural key often KONO + SUNO + DIVI for row‑level security

Finance Complexity

  • Data spread across header, line, distribution, ledger tables; no explicit FK constraints
  • Joins rely on shared fields (voucher no, journal no, KONO, DIVI, SUNO) and application logic
  • Need to join at correct grain, filter by latest version & posting status
  • Use Metadata Publisher’s Join Index & key order to guide joins

Salesforce Integration Overview

  • Objects synced from M3 → Salesforce: Accounts, Products, Price Books/Agreements, Orders, Order Lines, NAC Inventory
  • Sync direction mainly M3‑to‑SF; occasional updates back for orders
  • Integration built with DBAmp: SQL staging tables → DBAmp stored procedures → upserts into Salesforce
  • Custom fields store KONO/DIVI on Account/Product/Order for security mapping
  • Security: sales‑rep access driven by territories/role hierarchy aligned to KONO + DIVI boundaries

Price Supports Process

  • Entered in Salesforce by sales reps, then moved to a separate operational table (source of truth)
  • Table holds unique key (customer + product + KONO + DIVI) and fields for effective/expiration dates, approval status
  • Pricing applied manually later; not queried in real‑time at quote/order
  • Approval workflow currently manual; need clarification on downstream updates & financial reconciliation (Sarah Wahlberg)

Cadence & Scheduling

  • Nightly batch runs: NAC at 5 am, Inter‑Plastic at 6 am, MP at 7 am
  • Repeats every 4 hours until evening (last run ~7 pm)
  • Scheduling lives in SQL Agent / Airflow (to be confirmed)

Error Handling & Monitoring

  • Failures logged to an error table, email alerts sent to integration owners
  • Retry policy: immediate retry, then escalation after second failure
  • Current monitoring handled by the integration team; no dedicated Salesforce admin at present

Environment Management

  • Salesforce sandbox mapped to M3 test instance (M3 Cloud Test)
  • Credential rotation and sandbox refresh impacts integration mappings – need documented process
